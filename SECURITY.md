# 보안 정책 (Security Policy)

## 지원 정책

PackNine은 **최신 릴리즈 버전에 한해서만** 보안 패치를 지원합니다. 이전 버전을 사용 중이라면
보안 수정 사항을 적용받기 위해 최신 버전으로 업데이트해주세요.

| 버전 | 보안 패치 지원 |
|---|---|
| 최신 버전 | 지원 |
| 그 외 이전 버전 | 미지원 |

## 설계 단계에서부터 방어하는 취약점 클래스

PackNine은 압축 프로그램에서 반복적으로 발견되어 온 취약점 클래스를 사후 패치가 아니라
**설계 단계(도메인 계층의 `ArchiveSecurityPolicy`)**에서부터 방어하는 것을 목표로 합니다.

### 1. Zip Slip (경로 탈출)

압축 해제 시 아카이브 내부 엔트리 이름에 `../../` 같은 상대 경로나 절대 경로를 포함시켜
목적지 디렉터리 바깥의 임의 위치에 파일을 쓰는 공격입니다. 다양한 압축 라이브러리/도구에서
반복적으로 보고되어 왔으며(예: CVE-2018-1002200 계열을 포함해 여러 언어의 압축 라이브러리에서
유사 사례가 다수 존재), 아카이브를 신뢰하지 않고 다루는 모든 소프트웨어가 반드시 방어해야
하는 대표적인 취약점입니다.

PackNine의 대응:
- `ArchiveSecurityPolicy`가 압축 해제 대상 아카이브의 **모든 엔트리 경로를 정규화한 뒤,
  목적지 루트(destination root) 디렉터리 하위에 위치하는지**를 검증합니다.
- 실제 디스크 쓰기 이전에 전체 엔트리를 먼저 검사하는 all-or-nothing 방식을 사용하여,
  일부 파일이 이미 써진 뒤 취약한 엔트리가 뒤늦게 발견되는 상황을 방지합니다.
- 검증에 실패한 엔트리가 하나라도 있으면 `UnsafeArchiveEntryError`를 발생시키고 해제 작업
  전체를 중단합니다.

### 2. 압축 폭탄 (Decompression Bomb)

매우 작은 아카이브 파일이 해제 시 디스크/메모리를 고갈시킬 정도로 거대한 데이터를 생성하도록
설계된 공격입니다. 무심코 열었다가 시스템 자원이 소진되는 서비스 거부(DoS) 상황으로 이어질 수
있습니다.

PackNine의 대응:
- 엔트리별 압축비(compressed size 대비 uncompressed size 비율) 상한을 검사합니다.
- 아카이브 전체의 누적 해제 용량 상한을 검사합니다.
- 두 기준 중 하나라도 초과하면 해제를 거부하고 사용자에게 경고합니다.

### 3. 심볼릭 링크를 이용한 임의 경로 쓰기

아카이브 엔트리로 심볼릭 링크(또는 하드링크, 디바이스 파일 등 특수 파일)를 포함시켜, 해제
과정에서 링크를 따라가며 목적지 디렉터리 바깥에 파일을 쓰거나 덮어쓰게 만드는 공격입니다.

PackNine의 대응:
- 기본적으로 심볼릭 링크/하드링크/디바이스 파일 등 특수 엔트리 타입을 **거부**합니다.
- TAR 계열은 Python 3.12+에서 제공하는 `tarfile`의 `filter='data'`(PEP 706)를 함께 적용해
  표준 라이브러리 차원의 방어도 이중으로 적용합니다.

### 4. 구버전 RAR 파서의 원격 코드 실행 이력

과거 일부 RAR 압축 해제 도구가 사용하던 구형 네이티브 파서(예: UnACEV2.dll)에서 심각한
원격 코드 실행(RCE) 취약점(CVE-2018-20250 계열)이 발견된 사례가 있습니다. 이는 신뢰할 수
없는 아카이브를 파싱하는 네이티브 코드가 메모리 안전성 문제를 그대로 노출할 수 있음을
보여준 대표적인 사례입니다.

PackNine의 대응:
- **자체 RAR 네이티브 파서를 구현하지 않습니다.** 대신 `rarfile` 라이브러리로 헤더만 파싱하고,
  실제 압축 해제는 시스템에 설치된 신뢰할 수 있는 `unrar` 또는 `bsdtar`(libarchive) 외부
  바이너리에 위임합니다.
- 이를 통해 RAR 포맷 파싱의 공격 표면을 PackNine 코드베이스 밖으로 분리하고, 취약점 발견 시
  해당 외부 도구의 업데이트만으로 대응할 수 있도록 합니다.
- 외부 바이너리가 설치되어 있지 않은 경우 조용히 실패하지 않고 `ExternalToolMissingError`로
  명확한 설치 안내를 제공합니다.
- RAR **압축**은 포맷 자체가 비공개/상용 라이선스이므로 지원하지 않으며, 해제 기능만
  제공합니다.

### 5. 의존성 최신화 및 지속적 취약점 스캔

- `py7zr`, `rarfile` 등 핵심 압축 라이브러리는 `pyproject.toml`에 **최소 버전**을 명시하여
  알려진 취약점이 수정된 버전 이상만 사용하도록 강제합니다.
- CI(GitHub Actions)에서 `pip-audit`을 실행하여 의존성에 알려진 CVE가 있는지 주기적으로
  스캔합니다.

### 6. 네이티브 파서의 메모리 손상 취약점 (OOB Read/Write, 버퍼 오버플로우)

C/C++로 작성된 압축 포맷 파서가 아카이브 헤더의 길이·오프셋 값을 충분히 검증하지 않고 그
값으로 곧바로 버퍼에 접근하면, 공격자가 조작한 오프셋으로 버퍼 경계를 벗어난 메모리를
읽거나 써서(OOB, Out-Of-Boundary) 임의 코드 실행이나 크래시로 이어질 수 있습니다. 실제로
상용 압축 프로그램의 자체 네이티브 압축 라이브러리에서 길이 값 검증 미흡으로 인한 OOB
취약점이 다수 보고된 사례가 있으며, 이는 압축 프로그램 카테고리 전반에서 반복적으로
나타나는 취약점 클래스입니다.

PackNine의 대응 (구조적 방어):
- **PackNine은 어떤 압축 포맷에 대해서도 자체 바이너리 파서를 구현하지 않습니다.**
  ZIP은 CPython 표준 라이브러리 `zipfile`, TAR 계열은 표준 라이브러리 `tarfile`, 7Z는
  순수 Python 구현체인 `py7zr`을 사용하며, RAR은 위 4번 항목처럼 아예 외부의 신뢰할 수
  있는 바이너리(`unrar`/`bsdtar`)에 위임합니다. 코드베이스 어디에도 `struct.unpack`,
  수동 바이트 오프셋 연산, `ctypes` 기반 버퍼 접근이 존재하지 않습니다(CI에서 이 패턴을
  정적으로 검색해 회귀를 방지합니다).
- Python은 바이트/리스트 인덱싱이 경계를 벗어나면 메모리를 잘못 읽는 대신 항상
  `IndexError`/`ValueError` 예외를 던지는 메모리 안전(memory-safe) 언어이므로, C/C++
  네이티브 파서에서 흔히 발생하는 OOB 클래스의 취약점이 PackNine 자체 코드에서는
  구조적으로 발생할 수 없습니다.
- 다만 `py7zr`처럼 성능을 위해 C 확장을 사용하는 의존성은 여전히 존재하므로, 5번 항목의
  최소 버전 고정과 `pip-audit` 상시 스캔으로 해당 의존성의 취약점을 지속적으로 추적합니다.

### 7. 대량 엔트리를 이용한 자원 고갈 (DoS)

압축비/전체 해제 용량이 정상 범위 안에 있더라도, 극단적으로 많은 수의 엔트리(예: 빈 파일
수백만 개)나 비정상적으로 긴 엔트리 이름을 포함시켜 압축 해제 처리 자체를 오래 걸리게
하거나 파일시스템 리소스(inode 등)를 고갈시키는 방식의 공격도 가능합니다.

PackNine의 대응:
- `ArchiveSecurityPolicy.max_entry_count`(기본 100,000개)로 아카이브의 전체 엔트리 개수
  상한을 검사합니다.
- `ArchiveSecurityPolicy.max_entry_name_length`(기본 1,024자)로 비정상적으로 긴 엔트리
  이름을 사전에 거부합니다.

### 8. NTFS ADS(Alternate Data Stream) 삽입

아카이브 엔트리 이름에 콜론(`:`)을 섞어 `normal.txt:evil.exe` 형태로 만들면, Windows/NTFS
환경에서는 이것이 별도 파일이 아니라 `normal.txt`의 **대체 데이터 스트림**으로 해석되어
숨겨진 콘텐츠를 심는 데 악용될 수 있습니다. RAR 등 일부 포맷의 ADS 처리 과정에서 이런
유형의 취약점이 실제로 보고된 사례가 있습니다.

PackNine의 대응:
- `ArchiveSecurityPolicy`가 콜론을 포함한 엔트리 이름을 전부 거부합니다(정상적인 Windows
  드라이브 문자 표기는 별도 검사에서 이미 차단되므로, 이 검사는 그 외의 모든 콜론 포함
  이름을 대상으로 합니다).

### 9. MoTW(Mark of the Web) 미전파

웹 브라우저로 압축파일을 내려받으면 Windows는 그 파일에 "인터넷에서 받음" 표시
(NTFS `Zone.Identifier` 대체 스트림)를 남기고, 이 표시가 있어야 해제된 실행파일을 열 때
SmartScreen이 동작하고 Office 문서를 열 때 보호된 보기가 작동합니다. 압축 프로그램이 이
표시를 해제된 개별 파일에 옮겨주지 않으면, 압축 해제만으로 이런 보호 기제를 우회당할 수
있습니다.

PackNine의 대응:
- 압축 해제가 끝나면 원본 아카이브의 `Zone.Identifier`(있는 경우)를 해제된 모든 파일에
  자동으로 복사합니다(`packnine/infrastructure/motw.py`).
- ADS를 지원하지 않는 파일시스템이거나 원본에 애초에 표시가 없으면 조용히 아무 일도 하지
  않으며, 이 과정의 실패가 압축 해제 자체를 실패시키지 않습니다.

## 지원하지 않는 기능으로 인해 해당하지 않는 취약점 클래스

PackNine은 아카이브 압축/해제에 기능 범위를 한정하고 있어, 아래와 같은 부가 기능을 가진
경쟁 프로그램에서 보고된 취약점 클래스는 해당 기능 자체가 없어 원천적으로 노출되지
않습니다.

- 내장 이미지 뷰어(WebP/TIFF/PSD/JXR 등 이미지 포맷 파서) — 이미지 미리보기 기능 없음
- ISO/UDF, ISZ, CAB, ASAR, MSI(Compound) 등 디스크 이미지·설치 패키지 포맷 지원
- 클라우드 연동, 광고/원격 업데이트 확인 등 네트워크 기능

## 취약점 제보 방법

PackNine에서 보안 취약점을 발견하셨다면, 공개 Issue가 아니라 **GitHub Security Advisory를
통해 비공개로 제보해주세요.** 공개 Issue로 등록될 경우 패치 전에 취약점 정보가 노출되어
악용될 수 있습니다.

제보 시 다음 정보를 포함해주시면 대응에 큰 도움이 됩니다.

- 취약점 재현 절차 (가능하다면 최소 재현 아카이브 파일 포함)
- 영향받는 버전
- 예상되는 영향 범위 (예: 임의 경로 쓰기, 서비스 거부 등)
